# Magic-Chirp 文件开发目标

本文档用于明确每个文件最终要实现什么功能、开发到什么程度算完成。目标版本是一个完整可用的产品，要求登录、AI 识别、创建记录、图片上传、地图展示、详情页、编辑删除、权限、文件存储、数据库持久化形成闭环。

## 1. `server.py`

开发目标：成为整个后端服务的入口文件。

最终应实现：

- 创建 FastAPI 应用。
- 配置 CORS，允许前端访问后端接口。
- 挂载 `/uploads` 静态文件目录，让前端可以访问上传图片。
- 挂载所有业务路由：
  - `/api/auth`
  - `/api/identify`
  - `/api/records`
  - `/api/map`
- 服务启动时初始化数据库。

不应该包含：

- 注册登录的具体逻辑。
- 创建记录的具体逻辑。
- AI 识别的具体逻辑。
- 大量数据库操作。

完成标准：

- 执行 `python -m uvicorn server:app --reload` 可以启动服务。
- 打开 `/docs` 可以看到所有接口。
- `/uploads/...` 路径可以访问上传文件。
- 路由、静态文件、启动初始化、跨域配置都稳定可用。

## 2. `backend/src/modules/auth.py`

开发目标：实现用户身份认证。

最终应实现：

- 用户注册。
- 用户登录。
- 获取当前登录用户。
- 密码哈希。
- 密码校验。
- token 生成。
- token 解析。
- 南大邮箱后缀校验。
- 为其他模块提供 `get_current_user()`。

核心接口：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/auth/me`

完成标准：

- 用户可以注册账号。
- 用户可以用邮箱和密码登录。
- 登录成功后返回 token。
- 前端带 token 请求 `/api/auth/me` 可以拿到当前用户。
- 非南大邮箱注册会被拒绝。
- 认证逻辑可以被其他模块复用。

## 3. `backend/src/modules/records.py`

开发目标：实现观鸟记录的核心业务。

最终应实现：

- 创建观鸟记录。
- 接收鸟种、AI 候选、描述、经纬度、地点、时间、可见性。
- 接收并保存图片。
- 将记录写入数据库。
- 将附件信息写入数据库。
- 查询记录详情。
- 查询我的记录。
- 查询公开发布信息。
- 提供鸟类列表和地点列表供前端筛选。
- 编辑自己的记录。
- 删除自己的记录。
- 检查记录访问权限。

核心接口：

- `POST /api/records`
- `GET /api/records/public`
- `GET /api/records/options`
- `GET /api/records/{record_id}`
- `GET /api/records/mine`
- `PUT /api/records/{record_id}`
- `DELETE /api/records/{record_id}`

完成标准：

- 登录用户可以创建一条带位置的观鸟记录。
- 创建成功后，记录能在地图接口中出现。
- 记录详情可以显示鸟种、位置、作者、图片、AI 候选结果。
- 用户只能编辑、删除自己的记录。
- 私有记录不能被其他用户查看。
- 可以按鸟种、日期、发布人检索公开发布信息。
- 前端可以直接获取可选鸟类列表和地点列表。
- 创建、查询、编辑、删除记录都能正常工作。

## 4. `backend/src/modules/identify.py`

开发目标：实现鸟种识别接口。

最终应实现：

- 接收用户上传的图片。
- 校验图片格式。
- 调用 AI 识别服务。
- 返回 Top-N 鸟种候选。
- 每个候选包含鸟种名称和置信度。
- 支持 mock 兜底。

核心接口：

- `POST /api/identify`

完成标准：

- 上传图片后返回候选鸟种列表。
- AI 服务不可用时仍可以返回 mock 结果。
- 返回格式稳定，前端不需要关心背后是真 AI 还是 mock。
- 支持真实 AI 接口或稳定的本地推理实现。
- 失败时有兜底机制。

## 5. `backend/src/modules/map_records.py`

开发目标：为地图页面提供公开记录点数据。

最终应实现：

- 查询所有公开记录。
- 返回地图 Marker 所需字段。
- 支持按鸟种筛选。
- 支持按时间筛选。
- 后续可支持按地图视野范围筛选。

核心接口：

- `GET /api/map/records`

完成标准：

- 前端打开地图页后，可以通过该接口拿到公开记录点。
- 每条记录包含：
  - `id`
  - `bird_name`
  - `latitude`
  - `longitude`
  - `location_name`
  - `observed_at`
  - `cover_image_url`
  - `author_nickname`
- 私有记录不会出现在地图上。
- 地图公开记录查询可支持鸟种、时间筛选。

## 8. `backend/src/schemas/identify.py`

开发目标：定义 AI 识别接口的数据格式。

最终应实现：

- 单个鸟种候选 schema。
- AI 识别响应 schema。

建议包含：

- `BirdCandidate`
- `IdentifyResponse`

完成标准：

- AI 识别接口固定返回 `candidates`。
- 每个候选包含 `name` 和 `confidence`。
- 响应包含 `source`，用于区分 `mock`、`api`、`local_model`。
- AI 识别候选格式统一，便于替换实现。

## 9. `backend/src/config/settings.py`

开发目标：集中管理项目配置。

最终应实现：

- 数据库配置。
- 上传目录配置。
- token 密钥配置。
- token 过期时间配置。
- 邮箱后缀配置。
- 图片格式限制。
- 图片大小限制。
- AI mock 开关。
- AI API 配置，后续使用。

完成标准：

- 其他文件不要到处硬编码路径、密钥、上传限制。
- 修改配置时只需要改这个文件或 `.env`。
- 所有环境相关配置集中管理。
- 后续部署时只改配置，不改业务代码。

## 10. `backend/src/storage/files.py`

开发目标：实现上传文件的保存和校验。

最终应实现：

- 创建上传目录。
- 校验图片类型。
- 校验图片大小。
- 生成唯一文件名。
- 保存上传文件。
- 返回前端可访问的 URL。
- 后续可支持删除文件。

建议包含：

- `ensure_upload_dirs()`
- `validate_image(file)`
- `generate_file_name(original_filename)`
- `save_upload_file(file, subdir)`
- `delete_file_by_url(file_url)`

完成标准：

- 用户上传图片后，文件能保存到 `uploads/records/`。
- 接口返回的 `/uploads/...` URL 可以访问图片。
- 非图片文件会被拒绝。
- 文件存储机制可扩展到图片、视频、音频。
- 文件访问链接稳定。

开发目标：定义数据库表结构。

最终应实现：

- `users` 表结构。
- `bird_records` 表结构。
- `attachments` 表结构。
- 可选 `verification_codes` 表结构。

如果使用 SQLite 原生 SQL：

- 这里保存建表 SQL。

如果使用 ORM：

- 这里保存 ORM model。

完成标准：

- 数据库能初始化。
- 表字段和 `字段设计.md` 保持一致。
- 表之间关系清楚：
  - 一个用户有多条记录。
  - 一条记录有多个附件。
- 表结构与业务完全对应。

最终应实现：

- 查询我的观鸟记录。
- 查询公开地图记录。
- 编辑观鸟记录。
- 删除观鸟记录。
- 创建附件。
- 查询附件。
- 删除附件。

建议包含：

- `get_connection()`
- `init_db()`
- `create_user(...)`
- `get_user_by_email(...)`
- `get_user_by_id(...)`
- `create_record(...)`
- `get_record_by_id(...)`
- `list_records_by_user(...)`

- 从 mock 内存实现换成 SQLite 时，函数名和返回格式尽量不变。

- 上层业务不直接写 SQL。

## 15. 最终主流程完成标准

当这些文件都达到完成标准后，系统应该可以完成：

```text
用户登录
  -> 打开地图
  -> 上传图片识别鸟种
  -> 选择鸟种
  -> 获取定位或手动选点
  -> 创建观鸟记录
  -> 新记录出现在地图上
  -> 点击地图点进入详情
```

这就是完整可用版本最重要的交付目标。
