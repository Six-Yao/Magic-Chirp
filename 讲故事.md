# Magic-Chirp 项目故事

> 用于答辩汇报的项目叙事框架。从用户视角和开发者视角两个维度展开，帮助评委理解我们为什么做、做了什么、怎么做的。

---

## 一、一句话说清楚

**Magic-Chirp 是一个 AI 辅助的校园观鸟记录与发现平台——拍照即识别，识别即记录，记录即分享。**

---

## 二、用户视角：一个人看鸟，不如一群人一起看

### 2.1 故事的起点：你看到了一只鸟，然后呢？

想象你是南京大学的一名大一新生。

某天午后，你从教学楼出来，在路边的树上看到一只从没见过的鸟——灰褐色的羽毛，脖子上有一圈黑白相间的斑点，像一串珍珠。你觉得很好看，拍了张照片。

然后呢？

你把照片发到年级群里："求助万能的群友，这是什么鸟？"

消息很快被刷上去。有人回了句"好像是斑鸠"，但没人能确定。照片沉在了聊天记录里，那只鸟的名字你最终也没记住。你甚至忘了那天是几月几号、在哪个位置看到的。

**不只是你。这一周里，校园里有十几个人在不同的角落看到了同样或不同的鸟。** 其中有人拍到了北大楼树梢上的珠颈斑鸠，有人在操场边偶遇了戴胜，有人在图书馆门口的白头鹎窝下站了五分钟。但你们彼此不知道对方的存在。这些发现散落在各自的相册里，散落在没人回复的群消息里，散落在口口相传然后被遗忘的闲聊里。

**这个场景每天都在校园里发生。** 而它背后是三个环环相扣的问题：

| 痛点 | 说明 | 社交层面的后果 |
|---|---|---|
| **不认识** | 见到鸟不认识，拍照不知道发给谁问。 | 你无法向别人描述你看到了什么，也就无法分享。 |
| **记不住** | 即使偶尔知道了名字，也没有地方记录。 | 你没法告诉别人"我之前在这里也见过"，经验没法传递。 |
| **找不到同伴** | 你不知道校园里还有谁也喜欢看鸟。 | 明明有几十个潜在的观鸟同好，但你们永远擦肩而过——一个人看鸟是孤独的。 |

这三个问题的共同指向是：**观鸟在校园里一直是一件"一个人"的事。** 没有平台把一个个孤立的发现连接起来。

### 2.2 为什么现有工具解决不了

eBird、懂鸟——这些工具的核心能力是"帮你识别"和"帮你记录"。

但它们都是为**个人使用**设计的。你识别了一只鸟，记录在你的列表里；你识别了十只鸟，记录在你一个人的列表里。你和另一个也用了这些工具的人之间，**没有交集**。

一个刚对鸟产生兴趣的学生，打开这些 App 的第一反应是：
> "我不认识这只鸟，我该怎么记录？"
> "我一个人记录了又有什么意义？"

它们解决了"识别"问题，解决了"记录"问题，但没解决**"连接"问题**——连接人和鸟、连接人和人、连接人和这个地方。

### 2.3 Magic-Chirp 的答案：记录即分享，地图即社交

Magic-Chirp 的核心设计理念与现有工具截然不同：**每一次记录，默认就是对所有人的一次分享。**

```
拍照 → AI 自动识别鸟种 → 用户确认（或修改） → 提交 → 出现在所有人的地图上
```

你不需要专门去"分享"——记录本身就是分享。你不需要去"找同伴"——你的发现就是给同伴的信号。

这一设计带来了三层变化：

1. **你认识了它** — AI 在 2 秒内告诉你它是"珠颈斑鸠"。但更重要的是——**你知道，在你之前已经有 23 个人在校园里见到过它，其中北大楼附近最多。**

2. **你留下了一个标记** — 位置自动定位、时间自动填充，你的记录出现在校园地图上。**你不是在填一张只属于你的表格，你是在一张大家都能看到的地图上插了一面小旗。**

3. **你的标记成为别人的线索** — 另一个同学打开地图，看到了你刚留下的标记。他想："原来逸夫楼那边有珠颈斑鸠，明天我经过时也看一眼。"**他出门的动力，来自于你的发现。而你的下一次出门，可能也来自于他的。**

### 2.4 一个具体的用户故事

> 周三下午，小李从逸夫楼出来，在楼下的灌木丛里看到一只从没见过的鸟——头顶有扇形冠羽，翅膀黑白相间，嘴又细又长。他打开 Magic-Chirp，拍照，AI 在两秒内返回了候选：**戴胜 89%**。他点了确认，提交。
>
> 他不知道的是，同一时刻有三个人刷到了他的记录。
>
> 宿舍里的小王打开地图，看到逸夫楼多了个新标记。他点进去——"戴胜，逸夫楼下灌木丛，今天 15:30"。小王心想："我天天走逸夫楼，从没见过戴胜。"他把截图发到了鸟友群里："兄弟们，逸夫楼有戴胜！"
>
> 正在北大楼附近散步的小张也看到了这条记录。她两周前在北大楼拍到过一只戴胜，一直不确定是不是自己认错了。现在看到小李的记录，她翻出当时的照片，确认了那就是戴胜。她在小李的记录下面点了"我也见过"——这是她第一次在这个平台上确认：**有人和我看到了同一只鸟。**
>
> 而刚注册两天的女生小陈，本来觉得自己的记录太少、不好意思发。但看到地图上大家都在标记——北大楼的珠颈斑鸠、操场的白头鹎、逸夫楼的戴胜——她想："逸夫楼我每天路过，明天我也留意一下。"第二天，她真的在逸夫楼看到了那只戴胜，拍下了自己的第一张鸟种照片。

**四个互不相识的人，因为一只鸟在逸夫楼的出现，被连接到了一起。** 小王成了信息的传递者，小张得到了确认的满足感，小陈迈出了第一次记录的那一步。而小李从头到尾只做了一件事——拍了张照，点了"发布"。

### 2.5 观鸟社交的本质：共同绘制一张好奇心地图

Magic-Chirp 将观鸟重新定义为一种**轻量户外社交**——不需要组局、不需要认识人、不需要专门的装备和时间。

> 你不需要专门为观鸟出门。你只是在散步时多看了一眼树梢，在跑步经过操场时留意到一只不一样的鸟，在爬山时掏出手机对准树丛里的一抹颜色。

但因为你用的是 Magic-Chirp，**你那"多看一眼"的瞬间，变成了地图上一个永久的标记点。** 后来的人经过这里，也会多看一眼——因为地图上的标记告诉他们，"这里有好东西"。

这就是 Magic-Chirp 所理解的社交：
- 不是点赞、评论、转发——那些是**社交的形式**。
- 而是"你在这里见过，所以我也来看看"——这才是**社交的本质**：**共同的兴趣，通过共同的空间，形成共同的经验。**

校园地图上的每一个标记点，都是一个人留下的便签，对另一个人说：**"嘿，这里有一只鸟，来看看。"**

## 三、开发者视角：从 0 到 1 的工程决策

### 3.1 核心命题：降低"识别门槛"

**问题本质：** 一个什么都不懂的新手，怎么在最短时间内完成一次有意义的观鸟记录？

**我们的答案：** 把 AI 识别放在流程的最前端。用户不需要先"知道鸟的名字"再记录——拍完照，系统告诉你它可能是什么。用户只需要确认。

这个决策驱动了整个产品的交互设计：

- **"+"按钮是最显眼的元素** — 创建记录是最高优先级操作。
- **拍照即触发 AI 识别** — 不需要额外步骤。
- **位置和时间自动填充** — GPS 定位 + 当前时间，减少手动输入。
- **确认而非填写** — 用户在 AI 候选里选一个，而不是从零输入。

### 3.2 技术选型与架构设计

#### 为什么是 FastAPI + SQLite + React？

| 选择 | 理由 |
|---|---|
| **FastAPI** | 异步支持好（AI 识别需要轮询）、自动生成 API 文档（前后端协作）、类型安全（Pydantic schema 即文档） |
| **SQLite** | MVP 阶段零配置、零运维。SQL 语法标准，后续升级 PostgreSQL 只需改连接串 |
| **React + Vite** | TypeScript 类型安全、组件化适合 App Shell 架构、Vite 开发体验快 |
| **像素手绘地图** | 比接入高德/Mapbox 更有校园辨识度，加载更快，不依赖外部服务，且有独特的视觉风格 |

#### 架构原则

```
server.py          ← 入口，只做路由挂载 + CORS + 静态文件
  ├─ modules/      ← 业务逻辑，每个模块独立
  ├─ schemas/      ← Pydantic 模型，前后端字段契约
  ├─ storage/      ← 文件操作，与业务解耦
  └─ database/     ← 所有 SQL 的收口层，上层不写 SQL
```

核心原则：**每一层只做一件事。** modules 不写 SQL，database 不碰 HTTP，schemas 不碰业务逻辑。

### 3.3 关键技术挑战与解决方案

#### 挑战 1：AI 识别的异步轮询

懂鸟 API 的识别是异步的——上传图片后返回一个任务 ID，需要轮询获取结果。

**方案：** 使用 `httpx.AsyncClient` + `asyncio.sleep` 实现异步轮询，最多重试 6 次，每次间隔 0.5~1 秒。轮询超时返回空列表，前端可手动填写鸟种兜底。

**为什么不用 WebSocket？** MVP 阶段轮询足够。6 次轮询最多 6 秒，符合非功能需求中"AI 识别 ≤ 3s"的体验目标（实际情况通常在 2~3 次轮询内返回）。

#### 挑战 2：前端地图的聚合与展开

校园地图上有几十上百个标记点时，直接渲染会密集到无法点击。

**方案：** 自定义聚合算法（`createClusters`）——计算两点间像素距离，低于阈值则合并为一个数字标记。展开后有 CSS 动画的轨道扩散效果，外层和内层以不同的速度旋转，形成动态的"鸟儿散开"视觉效果。

**为什么不直接用高德地图 SDK 的聚合？**
- 对像素手绘风格的校园地图来说，高德的瓦片地图反而破坏设计感。
- 我们只需要一个简单的经纬度→像素位置的映射，自己实现更轻量。
- 动画效果完全可控，可以做出符合产品调性的交互。

#### 挑战 3：权限模型

记录有公开/私密两种可见性，详情页需要区分"游客/作者/他人"三种角色。

**方案：** 后端提供两个认证依赖——`get_current_user()`（强制登录）和 `get_optional_current_user()`（可选登录）。记录详情接口用后者，在业务层判断 `visibility == "private" && user_id != current_user.id`。

### 3.4 MVP 的边界取舍

**做了什么：**
- 核心闭环：注册 → 登录 → 拍照识别 → 创建记录 → 地图展示 → 查看详情
- 个人记录管理（查看、修改昵称/签名）
- 地图聚合展示 + 筛选

**没做什么（而且有意识不做）：**
- ❌ 首页信息流、点赞、评论、分享
- ❌ 关注/好友系统
- ❌ 鸟种图鉴百科
- ❌ 邮箱验证码

这些不是"没来得及做"，而是**阶段的明确取舍**。阶段一的目标是"让用户能完成一次完整的观鸟记录"，而不是"做一个功能齐全的社交平台"。社交功能放在后续阶段，第一阶段社交仅体现在"地图上看到他人记录"这一层。

## 四、项目的独特价值

### 4.1 与现有产品的差异

| | eBird | 懂鸟 | Magic-Chirp |
|---|---|---|---|
| 目标用户 | 专业观鸟者 | 观鸟爱好者 | **校园学生（含新手）** |
| 核心交互 | 表单记录 | 离线识别 | **拍照→AI识别→地图社交** |
| 社交属性 | 弱（数据共享） | 几乎无 | **地图即社交** |
| 入门门槛 | 高 | 中 | **低（30秒上手）** |
| 视觉风格 | 工具型 | 工具型 | **像素手绘 × 校园地图** |

### 4.2 为什么是大学校园

校园是一个天然适合"地图社交"的场景：

- **地理边界清晰** — 不需要处理全国范围的数据稀疏问题。
- **人群密度高** — 同一片校园里可能每天都有几十次观鸟机会。
- **社交意愿强** — 大学生天然希望发现同好。
- **鸟类多样性好** — 南京大学鼓楼、仙林校区及周边记录到的鸟类超百种。

Magic-Chirp 不是要做一个"所有人的观鸟平台"，而是做一个**南京大学学生的观鸟平台**。这个定位让它在 MVP 阶段就能获得足够的用户密度，而不需要像 eBird 那样依赖全球数据量。

## 五、项目进展与展望

### 5.1 当前状态

**阶段一 MVP 已完成：**

- ✅ 南大邮箱注册/登录（JWT 认证）
- ✅ AI 鸟种识别（Mock 兜底 + 真实懂鸟 API 异步识别）
- ✅ 观鸟记录创建（拍照 → AI 识别 → 确认鸟种 → 自动定位 → 发布）
- ✅ 像素校园地图（聚合标记、缩放拖拽、GPS 定位、地图点选）
- ✅ 记录详情（图片、鸟种、置信度、位置、时间、作者）
- ✅ 个人记录管理（查看、筛选、编辑资料）
- ✅ 移动端适配（响应式布局、触摸手势）

**进行中 / 短期待补：**

- 🔧 记录编辑功能（后端接口已有，前端待接入）
- 🔧 记录删除功能（"我的"页面待加删除按钮）
- 🔧 数据充实（更多鸟类大头照、options 数据）
- 🔧 性能优化（字体本地化、WebP 图片转换）

### 5.2 后续阶段展望

**阶段二 — 社交深化：**
- 首页信息流（关注用户的记录推送）
- 点赞/评论（轻量互动）
- 个人主页

**阶段三 — 知识沉淀：**
- 鸟种图鉴（基于集体记录自动生成）
- 校园鸟类排行榜
- 观鸟挑战/打卡

**阶段四 — 扩展：**
- 多校联盟（扩展到其他高校）
- 离线识别
- 语音识别（听鸟叫识别鸟种）

## 六、答辩叙事建议

### 6.1 推荐的讲述结构（5-7 分钟）

1. **开场（30秒）：一个场景**
   > "你在校园里看到一只不认识的鸟。你拍了张照，发到群里问。消息被刷上去了，没人回复。那只鸟的名字你最终也不知道。"
   >
   > "我们做了一个东西，让这个过程变成——拍照，然后你就认识了它。它叫珠颈斑鸠，而且你的发现会出现在校园地图上，成为下一个人出门的理由。"

2. **产品演示（2分钟）：完整走一个闭环**
   - 打开 App → 看到像素校园地图
   - 拍照 → AI 识别返回"珠颈斑鸠 86%"→ 确认
   - 自动定位 + 时间 → 发布
   - 地图上出现新的标记点
   - 点击标记点 → 查看详情

3. **技术亮点（1.5分钟）：挑 2-3 个讲**
   - 架构设计（分层解耦，2 周 4 人可完成）
   - AI 异步识别 + Mock 兜底
   - 前端自定义聚合地图

4. **项目价值（1分钟）：为什么值得做**
   - 降低观鸟门槛（从"我不认识所以不记录"到"不认识也能记录"）
   - 构建校园好奇心地图
   - 观鸟作为轻量户外社交的新载体

5. **展望（30秒）：后续计划**

### 6.2 可能的评委问题 & 应答方向

| 问题 | 应答方向 |
|---|---|
| "和懂鸟/eBird 有什么区别？" | 他们是专业工具，我们是校园社交。门槛更低，社交属性更强。 |
| "用户量不够怎么办？" | 校园场景天然有地理密度优势，MVP 只需要几百用户就能让地图"活"起来。 |
| "AI 识别不准怎么办？" | 用户可手动修改。AI 是降低门槛的工具，不是最终裁定者。且用的人越多，数据越多，未来可以微调模型。 |
| "怎么保证内容质量？" | 阶段一先做可见性控制（公开/私密）+ 管理员删除。阶段二考虑举报机制。 |
| "为什么不用高德地图？" | 像素手绘风格更有校园辨识度，加载更快，不依赖外部服务。技术上是轻量选择，体验上是差异化设计。 |

---

> **写在最后：** Magic-Chirp 做的事情本质上很简单——让一个对鸟一无所知的人，能在 30 秒内认识一只鸟，并且让这个发现帮助到下一个想看鸟的人。技术服务于这个目标，架构服务于两周的交付压力，设计服务于"降低门槛"这个核心命题。所有的取舍，都回到一个问题：**这件事对用户来说，是不是更简单了？**
