# Magic-Chirp 界面与交互逻辑设计

本文档基于当前后端结构和接口暴露方式设计完整界面与交互逻辑。核心原则是：地图是主入口，记录创建和记录查看都尽量在主界面弹窗内完成，减少页面跳转，让用户保持“在校园地图上发现鸟”的体验。

## 1. 后端接口对应关系

当前前端主要依赖以下接口：

| 场景 | 方法 | 接口 | 说明 |
|---|---|---|---|
| 注册 | `POST` | `/api/auth/register` | 创建用户 |
| 登录 | `POST` | `/api/auth/login` | 获取 token |
| 当前用户 | `GET` | `/api/auth/me` | 获取当前登录用户 |
| AI 识别 | `POST` | `/api/identify` | 上传图片并返回鸟种候选 |
| 创建记录 | `POST` | `/api/records` | 创建观鸟记录 |
| 我的记录 | `GET` | `/api/records/mine` | 获取当前用户记录 |
| 记录详情 | `GET` | `/api/records/{record_id}` | 获取单条记录详情 |
| 地图记录 | `GET` | `/api/map/records` | 获取公开记录点 |

需要登录的接口应带：

```text
Authorization: Bearer <access_token>
```

图片上传和创建记录使用 `multipart/form-data`。

## 2. 全局页面结构

应用采用移动端优先的固定 App Shell 结构。顶部标题栏、底部导航栏和底部中央 `+` 创建按钮保持固定，中间内容区根据当前底部导航项切换。

```text
AppShell
  顶部标题栏：当前视图标题、搜索 / 筛选 / 用户入口
  中间内容区：
    地图：校园地图、小鸟点位、记录预览弹窗
    记录：观鸟记录列表
    鸟种：鸟种图鉴或鸟种列表
    我的：用户窗台页、我的记录
  底部导航栏：地图 / 记录 / 鸟种 / 我的
  底部中央 +：创建记录入口
```

主界面切换逻辑：

- 点击底部“地图”：中间内容区显示校园地图和公开记录点。
- 点击底部“记录”：中间内容区显示记录列表。
- 点击底部“鸟种”：中间内容区显示鸟种列表或图鉴。
- 点击底部“我的”：中间内容区显示用户窗台页。
- 点击底部中央 `+`：未登录时打开登录弹窗，已登录时打开创建记录弹窗。
- 点击地图小鸟点位或记录列表项：打开同一个记录详情弹窗，不强制跳出当前 App Shell。

建议首屏进入逻辑：

- 未登录用户：可以进入地图页，浏览公开记录点和摘要弹窗。
- 未登录用户点击创建记录：弹出登录提示。
- 未登录用户点击记录详情：弹出登录提示或跳转登录页。
- 已登录用户：可以创建记录、查看详情、进入用户页。

## 3. 视觉风格

整体风格：

- 像素画风。
- 亲切、明媚、校园感。
- 主色：浅绿色、天空蓝、奶白色、嫩黄色。
- 地图、按钮、弹窗、图标都有轻微像素感。
- 页面必须保持可用性，不做纯装饰性游戏界面。

信息层级：

- 地图永远是主视觉。
- 弹窗面板只占需要的空间。
- 文字简短，重点突出鸟种、图片、位置、时间。

## 4. 页面一：登录 / 注册页

### 4.1 页面目标

让用户完成身份进入。注册和登录可以在同一页面通过 Tab 切换。

### 4.2 页面元素

- App 名称：Magic-Chirp / 神奇啾啾。
- 简短标语：发现校园里的小鸟。
- 邮箱输入框。
- 密码输入框。
- 昵称输入框，仅注册时出现。
- 登录按钮。
- 注册按钮。
- 进入地图看看，游客入口。

### 4.3 交互逻辑

登录：

```text
用户输入邮箱和密码
  -> 点击登录
  -> POST /api/auth/login
  -> 成功：保存 access_token，进入主地图页
  -> 失败：提示邮箱或密码错误
```

注册：

```text
用户输入邮箱、密码、昵称
  -> 点击注册
  -> POST /api/auth/register
  -> 成功：提示注册成功，引导登录，或自动登录
  -> 失败：提示邮箱格式、重复注册等错误
```

游客进入：

```text
点击游客浏览
  -> 进入主地图页
  -> 不保存 token
```

## 5. 页面二：主地图页

### 5.1 页面目标

主地图页是整个应用的核心。用户在这里发现公开观鸟记录，也可以创建自己的记录。

### 5.2 页面布局

```text
顶部栏：
  左侧：Magic-Chirp 标识
  中间：搜索或筛选入口
  右侧：用户头像 / 登录按钮

主体：
  校园地图
  小鸟头像点位 Marker

底部：
  中央固定 + 按钮
```

### 5.3 地图数据加载

进入地图页时：

```text
GET /api/map/records
  -> 返回公开记录点
  -> 前端根据 latitude / longitude 渲染 Marker
```

Marker 使用字段：

| 字段 | 用途 |
|---|---|
| `id` | 点击后查询详情 |
| `bird_name` | Marker tooltip 或摘要 |
| `latitude` | 地图纬度 |
| `longitude` | 地图经度 |
| `location_name` | 摘要地点 |
| `observed_at` | 摘要时间 |
| `cover_image_url` | 摘要图片 |
| `author_nickname` | 摘要记录者 |

### 5.4 小鸟点位视觉

每个点位显示为小鸟头像气泡：

- 圆形像素边框。
- 内部是鸟种头像或默认鸟图标。
- 有图片时可用 `cover_image_url` 的缩略图。
- 没图片时使用默认小鸟头像。
- 鼠标悬浮或点击时高亮。

### 5.5 点击小鸟点位

点击 Marker 后，不直接跳页，而是打开记录预览弹窗。

```text
点击 Marker
  -> GET /api/records/{record_id}
  -> 如果未登录且后端要求登录：显示登录提示
  -> 如果成功：打开记录预览弹窗
```

如果想减少请求，也可以先用 `/api/map/records` 返回的数据打开轻量弹窗，再在用户点击“查看详情”时请求详情。

推荐方式：

- 第一阶段点击 Marker 直接请求详情。
- 因为弹窗需要图片、AI 候选和附件，详情接口更完整。

## 6. 组件：记录预览弹窗

### 6.1 触发方式

用户点击地图上的小鸟点位。

### 6.2 展示方式

- 手机端：底部抽屉弹窗。
- 桌面端：右侧浮层面板。

### 6.3 面板内容

```text
顶部：关闭按钮
图片区域：用户上传图片
鸟种名称：如 珠颈斑鸠
记录者：author.nickname
时间：observed_at
地点：location_name
AI 识别：候选结果或最高置信度
按钮：查看完整详情 / 关闭
```

### 6.4 数据来源

接口：

```text
GET /api/records/{record_id}
```

使用字段：

| 字段 | 用途 |
|---|---|
| `bird_name` | 标题 |
| `author.nickname` | 记录者 |
| `observed_at` | 时间 |
| `location_name` | 地点 |
| `attachments` | 图片展示 |
| `ai_candidates` | AI 候选展示 |
| `latitude` / `longitude` | 小地图或定位说明 |

### 6.5 交互细节

- 点击关闭：弹窗关闭，地图保持当前视野。
- 点击查看完整详情：可以展开当前弹窗，或跳转到详情页。
- 未登录时点击详情：打开登录弹窗，登录后回到该记录弹窗。

推荐：前期直接让预览弹窗承载完整详情，不额外做详情页；如果内容变多，再拆独立详情页。

## 7. 组件：底部 `+` 创建记录入口

### 7.1 触发方式

用户点击主地图页底部中央 `+` 按钮。

### 7.2 登录判断

```text
点击 +
  -> 检查本地是否有 access_token
  -> 没有 token：打开登录提示
  -> 有 token：GET /api/auth/me 验证
  -> 验证成功：打开创建记录弹窗
  -> 验证失败：清除 token，打开登录提示
```

## 8. 组件：创建记录弹窗

### 8.1 展示方式

- 手机端：底部抽屉。
- 桌面端：居中弹窗或右侧面板。

### 8.2 表单步骤

创建记录建议设计成轻量分步流程：

```text
步骤 1：上传图片
步骤 2：AI 识别并选择鸟种
步骤 3：确认位置和时间
步骤 4：填写备注和可见性
步骤 5：发布
```

### 8.3 步骤 1：上传图片

页面元素：

- 上传框。
- 拍照按钮。
- 图片预览。

交互：

```text
用户选择图片
  -> 前端显示预览
  -> POST /api/identify
  -> 返回 candidates
  -> 进入鸟种选择状态
```

接口：

```text
POST /api/identify
Content-Type: multipart/form-data
field: image
```

### 8.4 步骤 2：AI 候选鸟种

展示：

```text
AI 猜测：
[珠颈斑鸠 86%]
[白头鹎 63%]
[乌鸫 41%]
[手动输入]
```

交互：

- 用户点击候选鸟种，填入 `bird_name`。
- 用户也可以手动输入鸟种。
- `ai_candidates` 原样保存，用于创建记录。

### 8.5 步骤 3：位置和时间

定位逻辑：

```text
打开创建弹窗
  -> 请求浏览器定位
  -> 成功：填入 latitude / longitude
  -> 失败：默认定位南京大学，提示用户手动选点
```

页面元素：

- 当前地点名称。
- 小地图缩略区域。
- 手动调整位置按钮。
- 时间选择器，默认当前时间。

字段：

| 字段 | 来源 |
|---|---|
| `latitude` | 浏览器定位或地图选点 |
| `longitude` | 浏览器定位或地图选点 |
| `location_name` | 逆地理编码或用户手填 |
| `observed_at` | 默认当前时间 |

### 8.6 步骤 4：备注与可见性

页面元素：

- 备注输入框。
- 公开 / 仅自己可见开关。

字段：

| 字段 | 说明 |
|---|---|
| `description` | 用户备注 |
| `visibility` | `public` 或 `private` |

### 8.7 步骤 5：发布

点击发布时：

```text
POST /api/records
  -> 成功：关闭创建弹窗
  -> 刷新 GET /api/map/records
  -> 新记录出现在地图上
  -> 可自动打开新记录预览弹窗
```

请求格式：

```text
multipart/form-data
bird_name
ai_candidates
description
latitude
longitude
location_name
observed_at
visibility
image
```

发布成功反馈：

- 地图出现新的小鸟点位。
- 显示轻量提示：“记录成功，新的小鸟点位已加入地图。”

## 9. 页面三：用户窗台页

### 9.1 页面目标

用户窗台页是个人信息和个人记录入口。它不是传统冷冰冰的个人中心，而是“我的观鸟小角落”。

### 9.2 页面布局

```text
左侧 / 上方：
  像素风窗台展示区
  用户头像
  昵称
  记录数量
  发现鸟种数量
  小鸟、植物、相机、望远镜等装饰物件

右侧 / 下方：
  我的上传记录列表
```

### 9.3 数据加载

进入页面时：

```text
GET /api/auth/me
GET /api/records/mine
```

使用字段：

| 接口 | 字段 | 用途 |
|---|---|---|
| `/api/auth/me` | `nickname` | 展示昵称 |
| `/api/auth/me` | `email` | 账号信息 |
| `/api/records/mine` | `bird_name` | 列表标题 |
| `/api/records/mine` | `cover_image_url` | 缩略图 |
| `/api/records/mine` | `location_name` | 地点 |
| `/api/records/mine` | `observed_at` | 时间 |
| `/api/records/mine` | `visibility` | 公开状态 |

### 9.4 我的记录列表

每条记录显示：

```text
缩略图
鸟种名称
地点
时间
公开 / 私密
```

点击记录：

```text
GET /api/records/{record_id}
  -> 打开记录详情弹窗
```

### 9.5 后续扩展

用户窗台页可以逐渐加入：

- 已发现鸟种图鉴。
- 用户称号。
- 收藏的小鸟。
- 常去地点。
- 观鸟日历。

但第一版不要让这些装饰影响记录列表的清晰度。

## 10. 页面四：记录详情

记录详情可以有两种实现方式：

### 10.1 推荐方式：弹窗详情

在地图页和用户页中都使用同一个记录详情弹窗。

好处：

- 交互轻。
- 地图上下文不丢失。
- 前端复用组件。

### 10.2 可选方式：独立详情页

路径示例：

```text
/records/{record_id}
```

适合后续分享链接、SEO 或更完整内容展示。

### 10.3 详情内容

详情应展示：

- 大图。
- 鸟种名称。
- 记录者。
- 观察时间。
- 地点。
- AI 候选结果。
- 用户备注。
- 小地图位置。
- 如果是本人记录，显示编辑和删除按钮。

## 11. 状态与异常处理

### 11.1 未登录

场景：

- 点击 `+`。
- 点击需要登录才能看的详情。
- 进入用户页。

处理：

```text
显示登录弹窗
登录成功后回到原操作
```

### 11.2 AI 识别失败

处理：

- 提示“识别失败，可以手动填写鸟种”。
- 保留图片。
- 允许继续创建记录。

### 11.3 定位失败

处理：

- 地图默认到南京大学。
- 提示“定位失败，请手动选择位置”。
- 用户点击地图选点。

### 11.4 图片上传失败

处理：

- 提示支持的图片格式和大小。
- 用户可以重新选择图片。

### 11.5 创建记录失败

处理：

- 保留表单内容。
- 展示错误原因。
- 允许重新提交。

## 13. 推荐主流程

### 13.1 看别人记录

```text
打开地图
  -> GET /api/map/records
  -> 渲染小鸟点位
  -> 点击点位
  -> GET /api/records/{id}
  -> 打开记录预览弹窗
```

### 13.2 创建自己的记录

```text
点击底部 +
  -> 检查登录
  -> 上传图片
  -> POST /api/identify
  -> 选择鸟种
  -> 获取定位或手动选点
  -> POST /api/records
  -> 刷新 GET /api/map/records
  -> 地图出现新点位
```

### 13.3 查看我的记录

```text
点击头像
  -> GET /api/auth/me
  -> GET /api/records/mine
  -> 展示窗台和记录列表
  -> 点击某条记录
  -> GET /api/records/{id}
  -> 打开详情弹窗
```

## 14. 最终交互结论

Magic-Chirp 的完整界面逻辑应是：

- 地图页是主界面。
- 小鸟头像点位负责发现。
- 点击点位打开记录弹窗，不强制跳页。
- 底部 `+` 号负责创建记录。
- 创建记录在弹窗内完成：图片、AI、位置、可见性、发布。
- 用户页做成窗台式个人空间，承载用户信息和记录列表。
- 所有核心操作都围绕当前后端接口组织，后端从 mock 换成真实数据库或真实 AI 时，前端交互不需要大改。
